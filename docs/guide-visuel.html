Le flow de bout en bout — ce qui se passe toutes les 30 secondes

1
Celery Beat (planificateur)
Déclenche dispatch_due_checks() toutes les 30 secondes
C'est le "chef d'orchestre" — il réveille le système à intervalle régulier. Il ne fait rien lui-même, il délègue.
2
dispatch_due_checks — tasks.py
Récupère la liste de tous les tenants actifs (ex: acme, globex…)
L'appli est multi-tenant : chaque client a son schéma PostgreSQL séparé. On boucle sur chacun.
3
_dispatch_tenant_checks — tasks.py
Pour chaque tenant : cherche les checks dont next_run_at ≤ maintenant
Un check "httpbin health" peut avoir un intervalle de 60s, un autre de 5min — on ne court que ceux qui sont dus.
4
_dispatch_tenant_checks — tasks.py
Met next_run_at = maintenant + intervalle avant de lancer la tâche
Technique anti double-dispatch : si deux workers lisent en même temps, le check ne tourne qu'une fois car le prochain tick est déjà planifié.
5
Celery Queue "checks"
Envoie run_check.delay(check_id, schema_name) dans la queue Redis
Chaque check devient une tâche indépendante — si httpbin met 5s, ça ne bloque pas le check de Google.
6
Worker Celery — run_check
Récupère la tâche depuis Redis, active le schéma tenant, appelle _execute_check
Le worker peut tourner sur un autre serveur — il a besoin du bon schéma PostgreSQL pour lire le bon Check.
7
_execute_check — tasks.py
Charge le Check depuis la BDD, appelle run_executor(type, config)
C'est ici qu'on sait si c'est un check HTTP, TCP ou Ping, et on appelle le bon exécuteur.
8
run_executor → executors.py (Bloc 1)
Exécute la vraie requête réseau : HTTP avec httpx, TCP avec asyncio, Ping avec icmplib
C'est la seule partie qui "sort" vers internet. Elle renvoie un ExecutorResult : statut + durée + code HTTP.
9
_execute_check — tasks.py
Sauvegarde un CheckResult en base (statut, durée, code HTTP)
L'historique complet est conservé. Utile pour les graphs de SLA et l'uptime sur 30 jours.
10
process_result — incidents.py (Bloc 3)
Analyse les derniers résultats et décide si un Incident doit être ouvert ou fermé
C'est la "mémoire" du système — un seul timeout n'alarme pas, mais 3 d'affilée oui.

State machine — quand un Incident s'ouvre, s'escalade et se ferme

Les 3 états possibles d'un Incident

OPEN
  • Incident détecté automatiquement
  • Personne n'a encore regardé
  • Alerte envoyée (futur Bloc 6)
ACKNOWLEDGED
  • Un humain l'a pris en charge
  • Défini via l'API (Bloc 4)
  • Peut laisser une note ack_note
RESOLVED
  • Le check repassé OK
  • Fermé automatiquement
  • resolved_at = heure de la récupération

Règles de transition (ce que fait process_result)

Pas d'incident
+ résultat FAIL/TIMEOUT mais pas encore atteint le seuil retry_count
Ex: 1 fail sur 3 requis → on attend
No-op
Pas d'incident
+ résultat FAIL/TIMEOUT et seuil atteint (consecutive ≥ retry_count)
Ex: 3 fails consécutifs sur 3 requis
→ OPEN
OPEN ACK
+ nouvelle failure
L'incident existe déjà, on escalade la sévérité si nécessaire mais on ne crée pas un doublon
No-op (sévérité mise à jour)
OPEN ACK
+ résultat OK
Le service répond à nouveau — on ferme automatiquement
→ RESOLVED
RESOLVED
+ nouvelle failure streak ≥ seuil
Le service est tombé à nouveau → nouvel incident, nouveau UUID
→ OPEN (nouveau)

Sévérité — comment elle est calculée

Failures consécutives Formule Sévérité
Entre 1× et 2× le seuil ex: retry_count=3 → entre 3 et 5 fails MEDIUM
Entre 2× et 3× le seuil ex: retry_count=3 → entre 6 et 8 fails HIGH
≥ 3× le seuil ex: retry_count=3 → 9 fails ou plus CRITICAL

Les 3 blocs — rôle et responsabilité de chacun

Bloc 1 — Executors
apps/monitoring/executors.py
Rôle unique : faire la vraie requête réseau et renvoyer un résultat.

Il ne sait pas qui l'appelle ni ce qu'on fera du résultat. Il répond juste :
→ status (ok / fail / timeout)
→ duration_ms (combien de temps ça a pris)
→ response_code (code HTTP si applicable)

3 types de checks :
• HTTP → httpx.AsyncClient
• TCP → asyncio.open_connection
• Ping → icmplib.async_ping
→
Bloc 2 — Tasks
apps/monitoring/tasks.py
Rôle : orchestrer — quand lancer les checks, pour quel tenant, et sauvegarder les résultats.

Il fait le lien entre la planification (Beat) et l'exécution (executors) :
1. Lit les checks dus en BDD
2. Met à jour next_run_at (anti double-dispatch)
3. Appelle run_executor (Bloc 1)
4. Sauvegarde le CheckResult
5. Appelle process_result (Bloc 3)
→
Bloc 3 — Incidents
apps/monitoring/incidents.py
Rôle : décider si la situation mérite une alerte ou si l'alerte peut être fermée.

Il reçoit chaque résultat et demande :
• Combien de fails consécutifs y a-t-il eu ?
• Est-ce qu'on a déjà un incident ouvert ?
• Le service vient-il de récupérer ?

→ Ouvre / escalade / ferme l'Incident en conséquence.
